## Database Normalization: Moving from 1NF to 3NF to eliminate anomalies

In database design, unorganized data leads to data redundancy and operational bugs known as insertion, update, and deletion anomalies. **Normalization** is the systematic process of decomposing complex tables to eliminate these inefficiencies. 

We systematically refine our data structure through three primary stages:
1. **First Normal Form (1NF):** Ensuring all table attributes are atomic (no multi-valued or repeating groups of data).
2. **Second Normal Form (2NF):** Meeting 1NF requirements and removing partial dependencies (ensuring all non-key columns depend entirely on the primary key).
3. **Third Normal Form (3NF):** Meeting 2NF requirements and eliminating transitive dependencies (ensuring non-key columns do not depend on other non-key columns).

```sql
-- Example of 3NF Design: Separating student data from department details
CREATE TABLE Departments (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(100)
);

CREATE TABLE Students (
    StudentID INT PRIMARY KEY,
    FullName VARCHAR(100),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);
```
##Data Cleansing: The parallel between normalization and machine learning pipelines

There is a striking parallel between database normalization and the Data Cleansing phase of machine learning. Raw, real-world datasets are inherently messy, plagued by duplicate rows, inconsistent features, and missing fields. Just as database normalization refines a relational structure to ensure data integrity, data preprocessing strips out anomalies before a predictive model ever sees the data. Feeding unorganized data into an algorithm results in skewed patterns; taking the time to carefully clean, normalize, and structurally organize data is what truly determines a model's operational success.
